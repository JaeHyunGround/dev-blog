## 1. 문제 정의

회사 프로젝트에서 하나의 애플리케이션 전체 코드베이스를 대상으로 리팩토링을 진행해야 하는 상황이 있었다.

기존 방식은 단순했다.

> 개발자인 내가 전체 코드를 직접 확인하고  
> 리팩토링이 필요한 부분을 판단한 뒤 수작업으로 개선한다.

하지만 코드 규모가 커질수록 다음 문제가 발생했다.

- 전 영역 직접 검토에 따른 시간 소모
- 리팩토링 필요 여부 판단의 주관성
- 중복 코드 및 구조 위반 요소 누락
- 생산성 대비 개선 효과 감소

특히 “리팩토링이 필요한지 판단하는 시간”이 가장 비효율적이었다.

이 과정을 자동화할 수 없을지 고민하게 되었고,  
AI 기반 멀티 에이전트 구조를 설계하게 되었다.

---

## 2. 설계 목표

단순 코드 수정 자동화가 목표가 아니었다.

- 코드베이스 전 영역 분석
- 리팩토링 필요 요소 도출
- 개선 설계 수립
- 안전한 코드 변경
- Human-in-the-loop 기반 검토 구조

이를 위해 역할을 분리한 멀티 에이전트 구조를 설계했다.

---

## 3. 전체 구조

```markdown
Orchestrator
├─ Analysis Agent (Claude Sonnet)
├─ Design Agent (Claude Sonnet)
└─ Execution Agent (Claude Opus)
```

### 역할 정의

| 구성 요소       | 역할                               |
| --------------- | ---------------------------------- |
| Orchestrator    | 전체 흐름 제어 및 단계별 검토 요청 |
| Analysis Agent  | 코드 분석 및 리팩토링 대상 도출    |
| Design Agent    | 분석 결과 기반 리팩토링 전략 설계  |
| Execution Agent | 설계 기반 실제 코드 변경 수행      |

---

## 4. 분석 기준

Analysis Agent는 다음 기준으로 전 영역을 스캔한다.

- 중복 코드 탐지
- FSD 레이어 규칙 위반 여부
- UI-로직 분리 필요성 (Custom Hook 분리 가능성)
- 타입 안정성 문제
- 복수 역할을 수행하는 컴포넌트 탐지
- 과도한 복잡도 컴포넌트 식별

단순한 감지가 아니라,  
왜 리팩토링이 필요한지 근거와 함께 리스트업하도록 설계했다.

---

## 5. 설계 단계

Design Agent는 분석 결과를 기반으로 다음을 정의한다.

- 파일 분리 전략
- Custom Hook 추출 구조
- FSD 레이어 재배치 방향
- 타입 구조 개선 전략
- 컴포넌트 책임 재정의

이 단계에서는 코드 변경을 수행하지 않는다.  
오직 설계만 수행한다.

---

## 6. 실행 단계

Execution Agent(Opus)는 설계 내용을 기반으로 실제 코드 변경을 수행한다.

실행 전 반드시 다음 정보를 출력한다.

- 변경 대상 파일 목록
- 변경 요약
- diff 형태 코드 제안

그리고 오케스트레이터는 개발자에게 묻는다.

> 다음 작업을 진행하시겠습니까?

- 승인 → 실행
- 수정 요청 → 설계 반영 후 재검토
- 중단 가능

완전 자동화가 아닌 단계별 승인 구조를 채택했다.

---

## 7. Human-in-the-loop 설계

AI를 전적으로 신뢰하지 않았다.

각 단계 종료 시 콘솔에:

- 분석 결과 요약
- 설계 전략 정리
- 코드 변경 diff

를 출력하고, 개발자가 진행 여부를 결정하도록 했다.

이 구조는 다음 리스크를 줄였다.

- 과도한 추상화
- 기존 의도 파괴
- 불필요한 구조 변경

---

## 8. 모델 분리 전략 및 토큰 최적화

각 에이전트에 서로 다른 모델을 사용했다.

- 분석 / 설계 → Claude Sonnet
- 실행 → Claude Opus

### 이유 1. 역할 특성 차이

- 분석·설계 단계는 구조적 사고와 맥락 이해가 중요
- 실행 단계는 코드 정확성과 일관성이 중요

### 이유 2. 토큰 최적화

코드베이스 전 영역을 다루기 때문에 토큰 사용량이 매우 크다.

- 분석 단계는 광범위한 입력 → Sonnet으로 비용 효율 유지
- 설계 단계는 요약된 분석 결과 기반 → Sonnet 유지
- 실행 단계는 제한된 파일 단위 작업 → Opus 사용

역할별 모델 분리는 비용과 정확성의 균형을 맞추기 위한 전략이었다.

---

## 9. 실제 사례: 무분별한 any 타입 제거

프로젝트 코드 내 일부 영역에서 `any` 타입이 과도하게 사용되고 있었다.

### 기존 코드 예시

```ts
function fetchUser(data: any) {
  return data.user.name;
}
```

문제점:

- 타입 안전성 상실
- 런타임 오류 가능성 증가
- IDE 자동완성 활용 불가

### 분석 에이전트 결과

- any 사용 위치 리스트업
- 타입 추론 가능 여부 분석
- 인터페이스 정의 가능성 제안

### 설계 제안

```ts
interface UserResponse {
  user: {
    name: string;
  };
}

function fetchUser(data: UserResponse) {
  return data.user.name;
}
```

### 결과

- 타입 안정성 강화
- 런타임 오류 가능성 감소
- 코드 가독성 개선

특히 사람이 놓치기 쉬운 여러 `any` 사용 지점을  
체계적으로 식별했다는 점이 유의미했다.

---

## 10. 도입 결과

멀티 에이전트 기반 AI 리팩토링 시스템 도입 이후 다음과 같은 변화가 있었다.

- 리팩토링 필요 요소 자동 탐지
- 중복 코드 제거
- UI-로직 분리 구조 개선
- 타입 안정성 강화
- 리팩토링 소요 시간 단축
- 동일 시간 대비 처리량 증가

특히 가장 큰 변화는

**“리팩토링이 필요한지 판단하는 시간”이 크게 줄어들었다는 점**이다.

기존에는 코드베이스 전 영역을 사람이 직접 검토해야 했다.  
이제는 분석 에이전트가 1차 후보를 도출하고,  
개발자는 검토와 의사결정에 집중할 수 있게 되었다.

### 11. 회고

AI는 리팩토링을 대체하지 않는다. 판단을 보조한다.
오케스트레이션 설계가 핵심이다.
Human-in-the-loop 구조는 반드시 필요하다.
멀티 에이전트 구조는 과하지 않았다. 오히려 안정성을 높였다.

### 12. 향후 확장 방향

- 자동 PR 생성 구조 확장
- 사내 공유 가능한 CLI 형태로 개선
- 테스트 코드 자동 생성 에이전트 추가
